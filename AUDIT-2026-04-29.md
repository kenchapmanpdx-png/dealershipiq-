## Critical

### C1. Internal worker bypass advertised but never honored
- `src/app/api/internal/sinch-process/route.ts:50-64` re-invokes `/api/webhooks/sms/sinch` with `x-internal-worker: 1` and `x-worker-secret`.
- `src/app/api/webhooks/sms/sinch/route.ts` never reads either header (grep: 0 matches for `internal|worker`).
- Result: when off-thread dispatch is enabled, every off-thread message will be silently rejected (no HMAC, no `to`-match → `unknown_format`).
- Fix: in webhook `POST`, top of handler, read `x-internal-worker` + `x-worker-secret`, timing-safe compare against `INTERNAL_WORKER_SECRET`, on match skip Sinch HMAC/XMS verification.

### C2. Dashboard auth will fail under Supabase P-256 migration
- `src/middleware.ts:230` calls `jwtVerify(tokenValue, secret)` with no `algorithms` option — accepts only HS256.
- Supabase migrated to ECC P-256 user-token signing ~2 months ago. Legacy HS256 secret only verifies anon/service_role.
- `verifyAndExtractClaims` returns null on alg mismatch → dashboard routes redirect logged-in managers to `/login` (line 140).
- Fix: switch to `createRemoteJWKSet(new URL('https://nnelylyialhnyytfeoom.supabase.co/auth/v1/.well-known/jwks.json'))` and pin `algorithms: ['ES256']`. OR drop local verify and read `app_metadata` from `getUser()` result already produced in `src/lib/supabase/middleware.ts:47`.
- Untested in this session — verify by attempting to load `/dashboard` as a manager. If redirected to `/login`, this is the cause.

### C3. Sinch webhook is fully synchronous — violates 15s budget
- `src/app/api/webhooks/sms/sinch/route.ts:277, 308` — `handleInboundMessage` awaited before 200; calls `gradeResponse` (4 attempts × 10s = 40s worst case per `src/lib/openai.ts:27`).
- Sinch retries at 15s → duplicate webhook deliveries → duplicate grading + duplicate SMS to user.
- Internal worker route (`/api/internal/sinch-process`) was built specifically to fix this but the webhook does not dispatch to it.
- Fix: in public `POST` handler, return 200 immediately after parse + auth, then `fetch(internalWorkerUrl, {...})` fire-and-forget. Combined with C1 fix, this restores the intended 2-hop architecture.

### C4. `/api/cron/grading-recovery` 500
Two issues, both real:

#### C4a. Likely missing runtime declaration
- `src/app/api/cron/grading-recovery/route.ts` has no `export const runtime` directive.
- Imports `crypto` (Node-only) via `src/lib/cron-auth.ts:5`.
- If Vercel auto-detects Edge for this route, module load throws.
- Fix: add `export const runtime = 'nodejs';` at top of file. 1-line fix.

#### C4b. `grading_started_at` is never written
- `src/lib/service-db.ts:190` `updateSessionStatus` only writes `{ status, updated_at }`.
- Webhook calls `updateSessionStatus(session.id, ..., 'grading')` (`webhooks/sms/sinch/route.ts:1242`).
- Cron filter `.eq('status','grading').lt('grading_started_at', cutoff)` excludes NULLs → cron always returns 0 candidates → never recovers stuck sessions.
- Fix: in `updateSessionStatus`, set `grading_started_at = now()` on transition to `'grading'`, set to `null` on `'completed' | 'error' | 'active'`.

### C5. `/leaderboard/demo-honda` 500
Two suspects, listed by likelihood:

#### C5a. PostgREST related-table filter ambiguity (likely)
- `src/app/api/leaderboard/[slug]/route.ts:68-88` does `.gte('training_results.created_at', ninetyDaysAgo)` on a nested 1:N relation without `!inner` join syntax.
- PostgREST raises on certain shapes when the related table is empty or filter is unresolved.
- Reproduce by hitting a brand-new dealership slug.
- Fix: drop the nested filter, fetch wider, filter in JS. Or invert — query against `training_results` as primary table.

#### C5b. Unhandled service-client throw (less likely, but check)
- `src/app/leaderboard/[slug]/page.tsx:31-62` is a server component with no try/catch.
- If `serviceClient` proxy throws on first `.from()` (e.g., transient network), error bubbles up as 500.
- Fix: wrap calls in try/catch, return `notFound()` on connection error.

### C6. INTERNAL_WORKER_SECRET missing — blocks C1 fix

- Once C1 is fixed and webhook starts dispatching, this var becomes critical.

### C7. STRIPE_* missing — `subscription-drift` cron will 500 every 12h

## High

### H1. No JWT algorithm pinning
- `src/middleware.ts:230` — `jwtVerify` accepts any alg the secret can verify.
- Defense-in-depth gap. Low standalone risk (HS256 secret can't verify EC).
- Fix: pass `{ algorithms: ['HS256'] }` (or `['ES256']` after C2 JWKS migration).

### H2. `x-subscription-required` written to RESPONSE not REQUEST
- `src/middleware.ts:177` sets the header on `response.headers`.
- `src/lib/auth-helpers.ts:85` `requireSubscription()` reads from `request.headers`.
- Header is invisible to route handlers AND leaks to client as info-disclosure.
- Currently no route actually calls `requireSubscription()` (grep: 0 matches), so the gate has been a no-op since added.
- Fix: set on `requestHeaders` (already mutated for `x-nonce` at line 93) OR delete `requireSubscription` and standardize on `checkSubscriptionAccess(dealershipId)` which all dashboard routes already call directly.

### H3. `/api/billing/checkout` creates orphaned auth user + dealership when Stripe is unconfigured
- `src/app/api/billing/checkout/route.ts:193-236` — rollback path runs after auth user + dealership rows are created.
- With Stripe envs unset, every signup attempt leaves orphaned data.
- Fix: short-circuit at top of route with 503 when `STRIPE_SECRET_KEY` / `STRIPE_PRICE_ID` missing. OR move Stripe call before any DB write.

### H4. Stripe webhook un-claim race
- `src/app/api/webhooks/stripe/route.ts:154-172` — on handler error, deletes `billing_events` row before returning 500.
- If DELETE fails (DB blip), row stays claimed forever, Stripe retry hits `{skipped:true}` indefinitely.
- A single unlucky retry can permanently drop `customer.subscription.deleted`, keeping a canceled customer billable.
- Fix: log to dead-letter queue or alerting on un-claim failure.

### H5. Cron pagination/budget gaps
- `subscription-drift` (`route.ts:24-27`) — unbounded `select('id, name, ...').not('stripe_customer_id', 'is', null)`. Will OOM at scale.
- `red-flag-check` (`route.ts:33-37`) — unbounded `select('id, name')` over all dealerships. No `createBudget`.
- `sync-optouts` (`route.ts:76-103`) — N+1 inside batches with serial `upsert` per membership. Vercel free tier 10s timeout.
- `daily-training` (`route.ts:111`) — `budget.shouldStop()` checked once before each dealership; inner `for (const user of eligible)` loop has no budget check. Add `if (budget.shouldStop()) break;` at line 177.
- Fix per: add `.limit(...)` and budget guards.

### H6. Scrambled subscription gate pattern
- `H8` middleware sets `x-subscription-required` header but no route reads it (H2).
- Routes (`team`, `sessions`, `gaps`, `coaching-queue`, `coach-themes`, `meeting-script`) call `checkSubscriptionAccess(dealershipId)` directly.
- `/api/ask/route.ts` skips the gate entirely — only checks `isFeatureEnabled('ask_iq_enabled')`.
- Unpaid dealership with feature flag on can use Ask IQ.
- Fix: either repair `requireSubscription` (per H2) and migrate all routes, OR delete `requireSubscription`, audit every protected route to call `checkSubscriptionAccess`, and make a code-review checklist item.

### H7. `next@14.2.35` is behind security patches
- 14.2.x line has had multiple security advisories since.
- Fix: `npm install next@14.2 --save-exact`, run tests, deploy.

### H8. NEXT_PUBLIC_VERCEL_ENV not set → Sentry events tagged `'development'` in prod
- `src/instrumentation-client.ts:70`, `sentry.server.config.ts:69`, `sentry.edge.config.ts:60` all fall back to `'development'`.
- Vercel does not auto-expose `VERCEL_ENV` as `NEXT_PUBLIC_*`.
- Effect: prod events filed under wrong environment in Sentry.
- Fix: set `NEXT_PUBLIC_VERCEL_ENV = production` in Vercel Production env. Or change server/edge configs to use non-public `process.env.VERCEL_ENV` directly.

### H9. `/api/coach/session` lacks `maxDuration`
- Calls GPT-4o (30s timeout per `route.ts:421`) plus DB fetches plus second GPT-4o-mini call.
- Vercel default timeout 10s on Hobby, 60s on Pro. Can 504 mid-OpenAI.
- Fix: `export const maxDuration = 60;` at top of file.

### H10. Migration drift
- 30+ migration files in `supabase/migrations/`, only 7 in `supabase_migrations.schema_migrations` table.
- Pre-2026-04-06 migrations applied directly via SQL editor. Schema is consistent.
- Risk: any branch / restore that replays from migrations table will skip 23 files.
- Fix: backfill the migrations table with `INSERT INTO supabase_migrations.schema_migrations (version, name) VALUES ...` for the 23 historical files. Or mark current schema as baseline.

### H11. SECURITY DEFINER + anon-callable functions
- `record_chain_step(p_chain_id, p_step, p_result)` — SECURITY DEFINER, callable by anon, does NOT check `auth.uid()`. Modifies `scenario_chains` based on caller-supplied IDs.
- `rls_auto_enable()` — SECURITY DEFINER, callable by anon (event trigger function but exposed to API).
- Fix: `REVOKE EXECUTE ON FUNCTION public.record_chain_step(uuid, integer, jsonb) FROM anon, authenticated;` and same for `rls_auto_enable()`.

### H12. 9 functions with mutable `search_path`
- `has_active_subscription`, `record_chain_step`, `unlock_user`, `try_lock_user`, `set_updated_at`, `validate_timezone`, `switch_active_dealership`, `get_dealership_id`, `get_user_role`, `is_manager`, `custom_access_token_hook`.
- SECURITY DEFINER without pinned `search_path` is a known privilege-escalation vector (caller's `search_path` controls function lookup).
- `custom_access_token_hook` runs on every login — highest priority.
- Fix: `ALTER FUNCTION public.<name>(<args>) SET search_path = public, pg_catalog;` for each.

### H13. Bootcheck doesn't include `INTERNAL_WORKER_SECRET`
- `src/lib/bootcheck.ts:35-65` — neither REQUIRED nor OPTIONAL_FUTURE.
- Once C1 is fixed (worker dispatch turns on), missing this var becomes a runtime crash with no deploy-time warning.
- Fix: add to OPTIONAL_FUTURE_ENV_PROD now; promote to REQUIRED when C1 ships.
