## Findings

### CRITICAL

**C1 — TEMP demo-emoji hack still live (committed 2026-04-30, never reverted)**
- `src/lib/openai.ts:209` — `export const DEMO_EMOJIS = true;` + call site :1037; allowlist `src/lib/sms.ts:53,78`.
- Every v7 grading SMS gets emoji prepended → forces UCS-2 encoding → 70-char segments instead of 160 → ~2-3x segments per grading message. Code comment at :207 says flip to false after demo.
- Cost impact modest at current volume but compounds with every rep added; also skews any per-segment cost tracking.
- **Fix (1 line):** set `DEMO_EMOJIS = false`. Optionally delete `applyDemoEmojis`, call site, and `DEMO_EMOJI_ALLOWLIST` filter in sanitizeGsm7.

**C2 — Off-thread Sinch dispatch is UNSAFE TO ENABLE as written — would convert lossless retry noise into silent message loss**
- `src/app/api/webhooks/sms/sinch/route.ts:291-311` + `src/app/api/internal/sinch-process/route.ts`. Dispatch gated on `SINCH_OFF_THREAD_ENABLED=true` + `INTERNAL_WORKER_SECRET`; neither set in Vercel. DO NOT set them until the code fix below lands.
- **Current inline behavior (dormant flag) is lossless:** slow grading → Sinch times out at 15s → Sinch RETRIES → DB idempotency dedupes → message eventually processed. Noisy (duplicate invocations, retry storms during OpenAI slowness) but no message is ever lost.
- **Off-thread path as written is lossy, three verified defects:**
  1. Webhook returns 200 to Sinch immediately after firing `void fetch()` (:298-310). A 200 means Sinch NEVER retries. The code comment ":289 Sinch will retry" is false in this path.
  2. No `waitUntil` anywhere in codebase. On Vercel the lambda freezes when the response completes; the un-awaited fetch (TLS handshake to own domain, ~tens of ms) frequently won't leave the machine before the `return` (microseconds later). Message dropped, no retry, `.catch` never executes on a frozen lambda.
  3. No `response.ok` check on the dispatch. A 401 (secret mismatch/typo across envs) or 500 from the worker hop resolves the fetch without throwing → zero logs, 100% silent drop.
- Also: worker route awaits the FULL re-entry (grading, 30s+) before returning (:56-64 of sinch-process), so hop-1 cannot simply `await` the dispatch — that recreates the 15s problem.
- Architecture note: the 3-hop bounce (Sinch → webhook → worker → webhook w/ `x-internal-worker` bypass) exists only because handlers live inside route.ts. Extracting `handleInboundMessage` to `src/lib/sinch-handlers.ts` and importing it in both routes removes a hop AND the HMAC-bypass header surface.
- **Fix order (code first, envs last):**
  1. `npm i @vercel/functions`; in webhook: `waitUntil(fetch(...).then(r => { if (!r.ok) log.error('sinch.webhook.off_thread_dispatch_rejected', { status: r.status }); }).catch(...))`.
  2. In sinch-process: validate secret → return 202 immediately → `waitUntil()` the re-entry fetch (so hop-1's tracked promise settles in ~100ms, not 30s). Or do the lib-extraction refactor and call the handler directly.
  3. Fix the false ":289 Sinch will retry" comment.
  4. Deploy, verify with a test SMS, THEN set `INTERNAL_WORKER_SECRET` (`node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`) + `SINCH_OFF_THREAD_ENABLED=true` in Vercel.
  5. Rollback lever: unset `SINCH_OFF_THREAD_ENABLED` → instantly back to lossless inline path.

### HIGH

**H1 — Template-fallback grades are silent**
- `src/lib/openai.ts` `getTemplateFallback()` — when both models fail (or circuit breaker open), rep receives a synthetic 3/5 grade indistinguishable from a real one. During an OpenAI outage, 100% of grades become meaningless with no dashboard signal.
- **Fix:** add `is_fallback boolean` to grading write + surface in dashboard; or send "grading delayed" SMS and let grading-recovery re-grade. Same applies to `generateFollowUp()` template fallback.

**H2 — Budget guards missing on 4 of 9 crons**
- Have `createBudget`: daily-training, red-flag-check, subscription-drift, sync-optouts. grading-recovery has an effective budget already (`.limit(200)` + compare-and-set updates) — no change needed there.
- Missing: daily-digest, dunning-check, challenge-results, orphaned-sessions. All have `maxDuration=60`, so failure mode = hard kill mid-loop, partial work.
- Worst: daily-digest (hourly, nested dealership→manager loops, no `.limit()`); near-UTC-midnight partial runs can double-send digests on retry.
- **Fix:** replicate the red-flag-check pattern (`createBudget` + `.limit()` + `markProcessed`) in the 5 routes.

**H3 — NOW keyword loop: orphaned sessions + inline execution**
- `src/app/api/webhooks/sms/sinch/route.ts:816-871` — per rep: create session → send SMS. If `sendSms` fails after session insert, rep has an 'active' session with no prompt SMS — dead-end until orphaned-sessions cron sweeps it (up to 2h).
- Also runs inside the inline webhook path today (see C2); 50-rep dealership × sequential sends with 50ms sleep can blow the 15s Sinch window.
- **Fix:** create session as `'pending'`, send SMS, then transition to `'active'` on send success (state machine already supports pending→active and pending→abandoned). Do NOT reorder to create-after-send — if send succeeds and the insert then fails, the rep holds a prompt whose reply can't route, which is a worse dead-end. Inline-execution pressure eases once C2's corrected fix lands (300s worker budget).

**H4 — Stripe webhook un-claim race (April H4, still outstanding, dormant)**
- Transient failure during event processing can leave the claim locked → retry silently skipped → subscription state drift. Low urgency until Stripe envs are set, but must land before billing launch. Fix: dead-letter on un-claim failure.

**H5 — `error` state is a dead-end for the rep**
- `src/lib/state-machine.ts:24` — `error → abandoned` only. A transient grading exception strands the session; no re-grade path (grading-recovery only rescues `status='grading'`), no user notification.
- **Fix:** either allow `error → active` with retry counter, or have orphaned-sessions send a "we hit a snag — reply NOW to restart" SMS when abandoning error sessions.

## Architecture Assessment

**Overall: sound.** The steel-thread shape is right — webhook → state machine → grader → SMS out, with recovery crons sweeping stuck states. RLS + service-role split is coherent; auth matrix showed no cross-tenant leakage. The April sprint fixed the right things.

**The single biggest improvement is completing C2 correctly — not flipping it.** The off-thread worker exists and inline execution is the root cause behind H3, M1's blast radius, and retry noise. But the dispatch as written trades lossless-with-retries for lossy-without-retries (see C2 defects). It needs the small `waitUntil`/`response.ok` fix (or the cleaner lib-extraction that removes the 3-hop bounce entirely) before the env vars are ever set. Enabling it as-is would be the worst single action available in this codebase.

**Second theme: silent degradation.** Template fallbacks (H1), TZ fallback (M8), fail-closed opt-outs (M9) all degrade invisibly. The pattern to adopt: every fallback writes a flag or Sentry event. The system currently fails gracefully but silently — fine until the day OpenAI has a 6-hour incident and a week of training data is synthetic.

**Third theme: unfinished sweeps, not design flaws.** Budget guards on 4/9 crons, zod on some routes, `.single()` vs `.maybeSingle()` — the right patterns exist in the codebase; they're just not applied everywhere. Each is a mechanical replication of an existing pattern.

**No rewrite candidates.** Module boundaries (state-machine, sms, openai, cron-budget, rate-limit) are clean. openai.ts is large (~1,300 lines) and could split into grader-v7 / grader-v6 / follow-up / fallbacks, but only when next touched — not worth a standalone refactor.
